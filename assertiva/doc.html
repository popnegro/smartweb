02 Lógica de Componentes (JS)

Assertiva utiliza el framework publicWidget de Odoo para encapsular la interactividad. Esto garantiza que los scripts solo se carguen cuando el DOM esté listo y el selector principal esté presente.

publicWidget.registry.AssertivaApp = publicWidget.Widget.extend({
    selector: '#wrapwrap',
    start: function () {
        return this._super.apply(this, arguments).then(() => {
            this.initRevealEffect();
            this.initIntlTelInput();
            this.initParallaxGlow();
            this.initSkeletonScreens();
            // ... motor de inicialización
        });
    }
});

07 Tokens & Vinculación

Se detalla cómo los valores definidos en user_values.scss permean hasta la estructura final en index.html.

Tipografía Dinámica

El valor $o-base-font-family: 'Noto Sans' viaja en el flujo:

SCSS: --font-heading: #{$o-base-font-family}
Regla: h1, h2 { font-family: var(--font-heading) }
HTML: <h1>Ciberseguridad...</h1>

Escala de Espaciado

Variables como $pt80 se vinculan a utilidades nativas:

Variable: $pt80: $o-spacing-unit * 10
Uso: Clase de utilidad .pt80 (Padding Top)
HTML: <section class="pt80"...>

Control de Animaciones

La variable $assertiva-reveal-duration (0.6s) define la velocidad global de entrada. En el HTML, la clase .assertiva-reveal activa este tiempo, pero la propiedad transition-delay se calcula dinámicamente en el SCSS para los grupos (nth-child), creando el efecto de "cascada" que se ve en la Hero y las Cards.

.assertiva-reveal { transition: opacity $assertiva-reveal-duration var(--anim-timing); }

08 Troubleshooting

Los cambios en JS/SCSS no se reflejan

Odoo concatena y minifica los archivos. Si los cambios no aparecen:

  • Activa el modo desarrollador con assets: ?debug=assets en la URL.
  • Usa la opción "Regenerate Assets Bundles" en el menú de depuración (icono de bug).
  • Limpia la caché del navegador o usa una ventana de incógnito.

El Widget no inicializa (No entra en start)

Comprueba los siguientes puntos críticos:

  • Asegúrate de que el archivo JS esté incluido en el bundle web.assets_frontend.
  • Verifica que el selector: '#wrapwrap' exista en el DOM.
  • Comprueba en la consola si hay errores de sintaxis previos que detengan la ejecución del registry.

Error 400 (Bad Request) al enviar formularios

Esto suele ser un fallo de validación CSRF.

// Solución implementada en scripts.js:
formData.append('csrf_token', odoo.csrf_token || request.csrf_token());

Validación de XML (QWeb)

Antes de subir cambios al editor de Odoo, valida la sintaxis XML:

# Ejecutar en terminal para verificar errores:
xmllint --noout ruta/al/archivo.html

Recuerda: El símbolo & debe escribirse siempre como &amp; (escapado) para evitar el error 500.

Depuración de Errores 500 (QWeb)

Si el servidor devuelve un Error 500 tras modificar el HTML:

  • Server Logs: Busca QWebException en la terminal de Odoo.
  • Flags de Dev: Inicia el servidor con --dev=xml para ver tracebacks detallados.
  • Cierre de Tags: Un </div (incompleto) es la causa número uno de errores críticos en Odoo 17.
  • Comillas: Verifica que no haya comillas dobles anidadas dentro de un t-attf-style.
Pro-tip: Usa console.log('DEBUG: ...') dentro de la función start() para confirmar que Odoo ha instanciado tu widget correctamente.

[Violation] Permissions policy: unload

Este aviso en la consola de Chrome es un comportamiento conocido en Odoo 17 causado por versiones heredadas de jQuery en el core. No es un error del código personalizado y no afecta la performance ni la persistencia de datos en el CRM. Puede ser ignorado con seguridad.

Cómo silenciar estos avisos en Chrome:

  1. Abre DevTools (F12) > pestaña Console.
  2. Haz clic en el icono de engranaje (⚙️) de la derecha.
  3. Desmarca la opción "Log violations".
  4. Esto ocultará el ruido de Odoo Core y permitirá ver tus logs de depuración claramente.

11 Mantenimiento del Código

Arquitectura Basada en Datos

El uso de bloques t-set en QWeb permite centralizar la información de servicios, soluciones y partners. Esto evita la duplicación de código (DRY) y reduce el riesgo de errores al actualizar enlaces o imágenes en múltiples lugares del HTML.

Ciclo de Vida JS

La implementación del método destroy en los widgets de Odoo garantiza que todos los intervalos de carrusel y oyentes (listeners) de eventos de scroll se limpien correctamente al navegar, evitando "memory leaks" y degradación de performance.

Limpieza de Markup

Se han eliminado etiquetas obsoletas (<font>) y estilos en línea (style="...") en favor de clases de utilidad. Esto permite que el editor de Odoo gestione el DOM sin generar duplicados accidentales durante la edición visual.

12 Escalabilidad y Crecimiento

El sistema está diseñado para expandirse sin requerir cambios estructurales profundos:

  • Nuevos Contenidos:

    Añadir una nueva tarjeta de solución o un logo de partner es tan simple como agregar una entrada al diccionario correspondiente en index.html. El grid y los carruseles se ajustan automáticamente.

  • Internacionalización:

    Al usar strings planos en las listas QWeb, Odoo los detecta como términos traducibles, permitiendo que el sitio escale a múltiples idiomas de forma nativa desde el backend.

13 Seguridad Web

CSRF Protection (A05)

Implementación de tokens únicos por sesión mediante request.csrf_token() para mitigar ataques Cross-Site Request Forgery.

XSS Prevention (A03)

Escape automático de caracteres vía QWeb y sanitización de inputs dinámicos en JS para prevenir Inyecciones.

Anti-Automation

Integración con Google reCAPTCHA v3 y límites de tasa (rate-limiting) nativos de Odoo para prevenir el spam masivo.

14 Performance Audit

LCP (Largest Contentful Paint)

Optimización del renderizado inicial del Hero:

  • Atributo fetchpriority="high" en imagen de fondo.
  • Implementación de Skeleton Screens para evitar el vacío visual.

CLS (Cumulative Layout Shift)

Eliminación de saltos visuales:

  • Dimensiones explícitas (width/height) en iconos SVG.
  • Uso de content-visibility: auto en el grid de soluciones.

Resultado: Mejora del TTI (Time to Interactive) al diferir el renderizado de elementos fuera del viewport.

15 Workflow & Validation

Para garantizar la estabilidad del servidor y evitar el error 500 causado por XML mal formado en Odoo, se ha establecido un flujo de validación obligatorio antes de cada despliegue.

Script de Validación XML (Python)

El archivo validate_xml.py utiliza la librería lxml para verificar la integridad del QWeb:

python3 validate_xml.py index.html

Criterios de Éxito:

  • Entidades &amp; escapadas correctamente en diccionarios t-value.
  • Cierre explícito de etiquetas no-vacías (evitar <span/>).
  • Ausencia de anidamiento de secciones que rompan el editor visual.
